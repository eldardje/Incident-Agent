3NF changes summary

INCIDENTS — removed 9 columns. Alarm metadata (alarm_name, metric_name, namespace, threshold, region, account_id) extracted to ALARM_SOURCES because the same CloudWatch alarm fires many incidents — storing alarm properties once eliminates redundancy and update anomalies. acknowledged_by, acknowledged_at, assigned_to removed because they are events, not static attributes — they already exist in INCIDENT_EVENTS and storing them twice causes inconsistency. resolution_time_minutes removed because it is a derived value (resolved_at - triggered_at) — computed at query time, never stored.

INCIDENT_CONFIG — removed 2 list columns. cloudwatch_log_groups and email_recipients were 1NF violations (multi-value attributes). Each is now a separate child table with one row per value, enabling independent add/remove operations and future per-row metadata (e.g. per-recipient severity thresholds).

AI_ANALYSIS — affected_services and fix_suggestions kept as DynamoDB List attributes. In a relational database these would be 1NF violations requiring child tables. In DynamoDB, storing them as lists is the idiomatic approach — they are always read as a whole with the parent record and are never individually queried or updated, so extracting them would add round trips with no normalisation benefit.

All tables — org_id present on every record. v1 value is always "default". Zero schema migration required when v2 multi-tenancy ships.
